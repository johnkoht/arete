# Soak Review Skill

Close the build→soak→review loop. A feature instrumented with the `## Usage Logging` pattern (PATTERNS.md) appends an objective per-run entry to `dev/soak/<feature>.md` in the workspace during a soak. This skill reads that log from BUILD, pairs it with the builder's gut reaction (deliberately NOT captured mid-run), and turns it into actionable findings.

**INTERNAL TOOLING** — For developing Areté itself, not for end users.

## When to Use

- "I've been soaking <feature>, check the logs."
- After a soak window, before deciding ship / iterate / roll back.
- Any time you want a structured read of a `dev/soak/<feature>.md` log.

Invocation: `soak-review <feature>` (e.g. `soak-review daily-winddown`). `<feature>` is the `<skill-id>` the log was written under.

## Paths (cross-repo)

- Log: `~/code/arete-reserv/dev/soak/<feature>.md` (workspace repo — read by absolute path).
- Live workspace skill copies: `~/code/arete-reserv/.arete/skills/{PATTERNS.md, <feature>/SKILL.md}`.
- Canonical skill source: `packages/runtime/skills/{PATTERNS.md, <feature>/SKILL.md}` (this repo).
- Findings output: `dev/work/plans/<feature-plan>/SOAK-FINDINGS.md` (this repo).
- Checkpoint: a `<!-- soak-review checkpoint: <ISO> -->` marker kept at the top of SOAK-FINDINGS.md.

## Workflow

### 1. Drift check FIRST — trust nothing until you've done this

The workspace `.arete/skills/` copies are synced **skip-if-exists** (`packages/core/src/services/workspace.ts:273-307`): once they exist, no `arete update`/install ever refreshes them. So the soak may have been generated by prose that no longer matches canonical. Before reading the log:

```bash
diff ~/code/arete-reserv/.arete/skills/PATTERNS.md            packages/runtime/skills/PATTERNS.md
diff ~/code/arete-reserv/.arete/skills/<feature>/SKILL.md      packages/runtime/skills/<feature>/SKILL.md
```

If they differ, **report the drift loudly** and treat affected log entries as suspect — note which canonical version the entries actually reflect. A forgotten post-merge `cp` is a finding-#15-class confounder one level up: data generated by code that no longer exists.

### 2. Read only the new runs

Read `dev/soak/<feature>.md`. Read the checkpoint marker in SOAK-FINDINGS.md (if present) and consider only entries newer than it — don't re-review runs already synthesized.

### 3. Gather the surrounding evidence

For the new entries: open the artifacts they point at (winddown docs, `raw-extractions/`, `dev/diary/*`), and read the arete-reserv `git log`/diff since the checkpoint. The log's `Anomalies` and `Agent notes` are leads, not conclusions — verify load-bearing claims against the real workspace data (don't trust a confidently-wrong note).

### 4. Harvest the subjective reaction

The capture file is objective by design. Cluster the new runs (by day, by config arm, or by anomaly) and ask the builder for a one-line gut reaction per run or cluster ("felt worse / no context / nailed it"). This is the highest-value signal and lives nowhere else.

### 5. Synthesize SOAK-FINDINGS.md

Group findings by severity (blocker / high / medium / low). For each, pair the objective record + the subjective reaction + your verified read. Propose fixes **cheapest-first** per the escalation rule: tier bump → prompt tighten → architecture. Flag any model-tier anomaly (a run that dropped to a weaker model) prominently. Distinguish "real bug" from "data point" from "works as intended."

### 6. Update the checkpoint

Write a fresh `<!-- soak-review checkpoint: <ISO> -->` marker so the next review starts after these runs.

## Boundaries

- Read-only against arete-reserv. No writes, commits, or index there.
- Don't fabricate fields: if an entry is malformed or missing the model tier, say so (and treat it as a possible silent-fallback run, not a clean one).
- This skill produces findings; it does not apply fixes. Hand the synthesized SOAK-FINDINGS.md back for the next build.
