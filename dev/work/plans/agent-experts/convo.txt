 lsp-pi is a great find. Here's what it gives us and why it matters for this plan:

 What it does: It gives agents an lsp tool — jump to definition, find references, hover for types, get diagnostics, list symbols.
 Basically IDE-level code navigation without reading entire files.

 Why it's valuable for expertise profiles: Right now, when an expert subagent needs to understand "what calls this function?" or
 "what's the type signature of this service?", it has to grep and read files. With LSP, it can do precise navigation:

 - Core expert assessing a change: lsp references → "these 14 call sites would break"
 - Developer implementing a task: lsp definition → jump straight to the interface
 - Reviewer checking a change: lsp diagnostics → type errors without running typecheck

 This makes the "dig deep" part of the expert workflow much more efficient. Instead of the profile needing to list every file to
 read, the expert can navigate the code graph directly.

 My recommendation: Install it — it's a clear upgrade for subagent quality. But it's orthogonal to the agent refactor. I'd install
 it now (takes 2 minutes) and let it benefit the current system immediately, then the new expertise model benefits from it
 automatically.

 Now, on the self-improving documentation point — I want to make sure we scope this right. There are two levels:

 1. Light maintenance (every expert run): Update LEARNINGS.md, flag profile inaccuracies. This is cheap and should be default
 behavior.
 2. Profile evolution (periodic): The expert reflects — "I kept needing to look up how the search index interacts with the template
 resolver, and my profile doesn't mention that relationship." It adds a cross-reference to its profile. This is more significant —
 it changes the map for future runs.

 Both are valuable. The question is whether profile evolution happens inline (expert updates its own profile at end of session) or
 as a dedicated maintenance task (planner spawns expert in "maintenance mode" periodically). I lean toward inline for small updates
 (add a cross-reference, update a file path) and dedicated for structural changes (reorganize profile sections, graduate
 LEARNINGS.md entries into profile knowledge).
