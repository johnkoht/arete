## How This Works

`packages/runtime/rules/` contains **PRODUCT rules** â€” rules that get copied into end-user workspaces during `arete install`. They are distinct from **BUILD rules** (`.cursor/rules/*.mdc`), which govern AretĂ© development. The runtime rules are stored in two subdirectories: `cursor/` (`.mdc` extension, used as-is for Cursor IDE) and `claude-code/` (also `.mdc` extension at commit time â€” renaming to `.md` happens at install time via `adapter.ruleExtension`). Both directories contain the same 7 rule files. The canonical source for content is `cursor/` â€” when adding or editing rules, edit `cursor/` first, then mirror to `claude-code/`. Both directories use `.mdc` extension and contain identical content at commit time. At install time, `workspace.ts` copies rules and renames files using `adapter.ruleExtension` (`.mdc` for Cursor, `.md` for Claude Code). The `ClaudeAdapter` in `packages/core/src/adapters/claude-adapter.ts` has a `transformRuleContent()` method that can convert `.cursor/` paths â†’ `.claude/` and `.mdc` â†’ `.md`, but this is NOT invoked during install â€” content is copied as-is. The rule files are: `agent-memory.mdc`, `arete-vision.mdc`, `context-management.mdc`, `pm-workspace.mdc`, `project-management.mdc`, `qmd-search.mdc`, `routing-mandatory.mdc`.

## Gotchas

- **Never write "either/or" path patterns in rule content.** Writing `.cursor/tools/X or .claude/tools/X` in a rule produces `.claude/tools/X or .claude/tools/X` after `transformRuleContent()` runs â€” the `.cursor/` half is transformed, leaving a broken duplicate. This is exactly what happened in `2026-02-13_multi-ide-path-fix.md`: a tool routing change added `or .claude/` alternatives in `pm-workspace.mdc` and `routing-mandatory.mdc`, breaking 5 path patterns. The fix was reverting to single `.cursor/` paths in all rules.

- **`transformRuleContent()` applies three transforms in order: `.cursor/skills/` â†’ `.agents/skills/`, then `.cursor/` â†’ `.claude/`, then `.mdc` â†’ `.md`.** The skills prefix is handled separately and takes priority â€” `.cursor/skills/X` becomes `.agents/skills/X`, NOT `.claude/skills/X`. See `claude-adapter.ts` L63â€“67.

- **Always use `.cursor/` paths in rule content â€” never `.claude/`.** `ClaudeAdapter.transformRuleContent()` transforms `.cursor/` â†’ `.claude/` automatically. If you hardcode `.claude/` in a rule, Claude Code users get it twice (both the original `.claude/` and a no-op transform of any adjacent `.cursor/` path). Correct: `.cursor/tools/onboarding/TOOL.md`. Broken: `.cursor/tools/X or .claude/tools/X`. From `2026-02-13_multi-ide-path-fix.md`.

- **`claude-code/` contains a SUBSET of `cursor/` rules.** After the Claude Code workspace consolidation, Claude Code uses only 3 rules (`agent-memory.mdc`, `context-management.mdc`, `project-management.mdc`) vs Cursor's 7. The remaining guidance (vision, routing, qmd-search, pm-workspace) is consolidated into `CLAUDE.md` generated by `packages/core/src/generators/claude-md.ts`. Both directories still use `.mdc` extension at commit time; renaming to `.md` happens at install time via `adapter.ruleExtension`. When adding a new rule for Claude Code: add it to `claude-code/` AND update the `getProductRulesAllowList('claude')` in `workspace-structure.ts`.

- **BUILD rules (`.cursor/rules/*.mdc`) and PRODUCT rules (`packages/runtime/rules/`) are different audiences â€” do NOT confuse them.** BUILD rules enforce quality gates, testing, and pre-mortem workflows for AretĂ© contributors. PRODUCT rules guide AI agents in end-user workspaces doing PM work. From `2026-02-12_rules-architecture-refactor-learnings.md`: "These serve different audiences and should not be confused." Before editing, confirm which audience your change targets.

- **The 5 old BUILD rules (`pm-workspace.mdc`, `routing-mandatory.mdc`, `qmd-search.mdc`, `context-management.mdc`, `project-management.mdc`) were deleted from `.cursor/rules/` in the 2026-02-12 refactor.** Those names now exist only in `packages/runtime/rules/` (PRODUCT side). If you see a reference to them in `.cursor/rules/`, it's stale â€” they were removed. From `2026-02-12_rules-architecture-refactor-learnings.md`.

- **Rule content changes must be mirrored to `claude-code/` for shared rules.** There is no build step that auto-generates `claude-code/` from `cursor/`. For the 3 rules that exist in both (`agent-memory`, `context-management`, `project-management`), editing only `cursor/` leaves `claude-code/` out of sync. Rules that exist only in `cursor/` (e.g., `routing-mandatory`, `qmd-search`) do not need mirroring â€” Claude Code consolidates that guidance into `CLAUDE.md`.

- **Skill rename `onboarding` â†’ `getting-started` (2026-02-22)**: The onboarding _skill_ (AretĂ© workspace setup) was renamed to `getting-started` to eliminate a naming collision with the onboarding _tool_ (30/60/90 job onboarding). When updating rules: "onboarding" in `pm-workspace.mdc` and `routing-mandatory.mdc` mostly refers to the **tool**, not the skill. The skill references were changed to `getting-started`; tool references (`.cursor/tools/onboarding/TOOL.md`, intent table entries, example queries) were preserved. **Lesson**: When a name is overloaded across skills, tools, and PM concepts, categorize each occurrence before changing. The pre-mortem Risk #3 flagged this and prevented accidental tool reference changes.

## Invariants

- Rule files in `cursor/` must contain only `.cursor/` path references â€” never `.claude/`, never `or .claude/` alternatives.
- `claude-code/` contains a subset of rules from `cursor/` (currently 3 of 7). Files that exist in both directories have identical content and use `.mdc` extension. Extension renaming happens at install time.
- The `claude-code/` file list is controlled by `getProductRulesAllowList('claude')` in `workspace-structure.ts`. Adding a rule to `claude-code/` requires updating both the directory and the allow list.

## Testing Gaps

- No automated test that runs `transformRuleContent()` over all `cursor/` rule files and asserts no broken path patterns (e.g. double `.claude/` references). The 2026-02-13 incident was caught by code review, not a test.
- Pre-commit check exists in `dev.mdc` (`rg "\.cursor.*or.*\.claude"`) but it's a manual step, not automated.

## Patterns That Work

- **Pre-commit grep before any rule edit**: `rg "\.cursor.*or.*\.claude|\.claude.*or.*\.cursor" packages/runtime/rules/` catches "either/or" patterns before they reach commit. From `dev.mdc` Multi-IDE Consistency Check section (added after `2026-02-13_multi-ide-path-fix.md`).
- **Edit `cursor/` first, then apply transforms to `claude-code/`**: Treat `cursor/` as the single source of truth and derive `claude-code/` mechanically.

## Pre-Edit Checklist

- [ ] Run `rg "\.cursor.*or.*\.claude|\.claude.*or.*\.cursor" packages/runtime/rules/` â€” must return no matches before AND after your change
- [ ] Confirm you're editing PRODUCT rules (`packages/runtime/rules/`) not BUILD rules (`.cursor/rules/`)
- [ ] Update BOTH `cursor/` and `claude-code/` for shared rules (agent-memory, context-management, project-management) â€” content must be identical for files that exist in both
- [ ] Run `npm run typecheck` and `npm test` to confirm no TS or test regressions
- [ ] If rule changes affect `arete install` behavior: test by running `arete install` in a temp directory
